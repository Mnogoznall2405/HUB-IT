## 1. Backend: LDAP и сервис

Файл: [`backend/services/ad_users_service.py`](C:\Project\Image_scan\WEB-itinvent\backend\services\ad_users_service.py)

### 1.1 Список OU (дерево)

Новая функция `list_ad_organizational_units(*, parent_dn: str | None = None)`:

- `parent_dn` пустой → дети корня домена (`_resolve_ad_search_base()`), scope `ONELEVEL`
- `parent_dn` задан → дети выбранного OU, scope `ONELEVEL`
- LDAP-фильтр: `(objectClass=organizationalUnit)`
- Атрибуты: `distinguishedName`, `name`, `ou`
- Ответ: `[{ dn, label, has_children }]`
- `has_children` — быстрая проверка `ONELEVEL` под OU (или эвристика «всегда true» на первом этапе, если проверка дорогая)

### 1.2 Отчёт по истечению паролей

Новая унифицированная функция `get_ad_password_expiry_report(*, ou_dn: str | None, mode: Literal["all","expiring"], days_threshold: int, q: str)`:

- **`ou_dn`** — `search_base` для LDAP (если не задан — корень домена, как сейчас)
- **`mode=all`** — переиспользовать логику [`get_ad_users_password_status()`](C:\Project\Image_scan\WEB-itinvent\backend\services\ad_users_service.py) (включая маппинг филиалов из IT-Invent), но с параметризованным `search_base`
- **`mode=expiring`** — переиспользовать фильтры из [`list_ad_users_expiring_soon()`](C:\Project\Image_scan\WEB-itinvent\backend\services\ad_users_service.py) (исключение служебных аккаунтов, `days_threshold`), тоже с `search_base=ou_dn`
- Добавить в LDAP-атрибуты `distinguishedName` (для отображения пути OU в таблице, опционально)
- Параметр **`q`** — клиентский или серверный поиск по `login` / `display_name` / `department` (начать с серверной фильтрации в Python после LDAP, как в других AD-функциях)
- Вернуть структуру:

```python
{
  "status": "ok",
  "ou_dn": "...",
  "mode": "all|expiring",
  "threshold_days": 7,
  "policy_days": 40,
  "total": 123,
  "users": [...]
}
```

Рефакторинг: вынести общий LDAP-поиск пользователей в `_search_ad_users(search_base, filter_extra=None)` чтобы не дублировать bind/unbind.

### UI-поведение

**Левая колонка (desktop) / Drawer (mobile):**
- Дерево OU с lazy expand
- Пункт «Весь домен» (корень) — `ou_dn` не передаётся
- Выбранный OU подсвечивается

**Верхняя панель:**
- `ToggleButtonGroup`: «Все пользователи» / «Только истекающие»
- При режиме «Истекающие» — `Slider` или `TextField` для `days_threshold` (default 7, как в AI tool)
- Поиск по ФИО / логину
- Кнопка «Обновить»

**Таблица (колонки):**
- ФИО (`display_name`)
- Логин (`login`)
- Отдел (`department`)
- Филиал (`branch_name`) — из существующего маппинга
- Дата смены пароля (`pwd_last_set_date`)
- Истекает (`expiration_date`)
- Осталось дней (`days_to_expire`)
- Статус: Chip — «Просрочен» / «Скоро» / «OK» / «Сменить сейчас»

**Сортировка по умолчанию:** как в backend — просроченные первыми, затем по `days_to_expire`.

**Состояния:** loading skeleton, пустой OU, ошибка LDAP (Alert), «LDAP не настроен».

## Риски и ограничения

- **Производительность:** режим «Все» на корне домена может вернуть тысячи записей. OU-фильтр снижает нагрузку; при необходимости позже добавить server-side pagination.
- **Политика 40 дней:** сейчас фиксированная константа `AD_PASSWORD_MAX_AGE_DAYS`, не реальная доменная политика AD. Менять не планируем в этом этапе.
- **Права:** страница остаётся за `passwords.read` в навигации; новые AD-endpoint тоже на этом permission (не admin-only).
