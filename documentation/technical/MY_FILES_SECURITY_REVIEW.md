# Security review: «Мои файлы»

Дата проверки: 2026-06-04

Область проверки: загрузка, обработка, хранение, скачивание, удаление и публичные
ссылки раздела «Мои файлы».

## Итог

Критического обхода авторизации или возможности угадать публичную ссылку не
обнаружено. Проверки владельца, срок действия файлов, одноразовые download-grant
токены и production-настройки реализованы корректно.

Риски P1, найденные при первоначальной проверке, закрыты и развёрнуты в
production 2026-06-04. Оставшиеся пункты P2/P3 относятся к разграничению прав,
аудиту, защите bearer-токенов и дополнительному усилению инфраструктуры.

## Статус исправлений P1

- Загрузка сначала создаёт атомарный quota-reservation со статусом `uploading`;
  тело запроса принимается только после проверки квоты, конкурентности и общей
  очереди.
- Для загрузок действуют отдельные rate limits по пользователю и IP, включая
  запросы из внутренней сети.
- Обработка вынесена из FastAPI в PM2-процесс `itinvent-my-files-worker`.
  PostgreSQL advisory-lock не допускает одновременный запуск нескольких worker,
  ZSTD ограничен двумя потоками, PM2 контролирует память процесса.
- Microsoft Defender работает в fail-closed режиме. Файл нельзя скачать или
  опубликовать до результата `clean`.
- Существующие файлы автоматически пересканированы; ZSTD-файлы перед проверкой
  распаковываются во временный карантин и затем удаляются.
- Alembic `20260604_0044` хранит статус, движок и время проверки.

## Подтверждённые защиты

- Все операции владельца используют `get_current_active_user`, а сервис
  дополнительно проверяет `owner_user_id`
  (`WEB-itinvent/backend/api/v1/my_files.py:95`,
  `WEB-itinvent/backend/services/my_files_service.py:627`).
- Публичный share-token и одноразовый download-grant создаются через
  `secrets.token_urlsafe(32)`, а поиск публичной ссылки выполняется по SHA-256
  хешу (`WEB-itinvent/backend/services/my_files_service.py:673`,
  `WEB-itinvent/backend/services/my_files_service.py:700`,
  `WEB-itinvent/backend/services/my_files_service.py:777`).
- Недействительные, истёкшие, удалённые и неготовые файлы не выдаются публично.
- Download-grant потребляется атомарно и только один раз.
- Имена файлов не используются для формирования физического пути; blob-пути
  строятся по SHA-256.
- Ответ скачивания использует `Content-Disposition: attachment`
  (`WEB-itinvent/backend/api/v1/my_files.py:84`).
- Размер одного файла ограничен 1 GiB на уровне приложения и IIS
  (`WEB-itinvent/backend/services/my_files_service.py:29`,
  `WEB-itinvent/backend/api/v1/my_files.py:64`,
  `WEB-itinvent/frontend/public/web.config:7`).
- Для публичных ссылок и download-grant включены rate limits с хранением
  счётчиков в AppDB.
- Проверенный runtime работает в `production`, с `debug=false`,
  `Secure + SameSite=Strict` cookie, CORS только для `https://hubit.zsgp.ru`,
  runtime rate-limit storage — AppDB.

## P1: исходные findings, закрыты 2026-06-04

### 1. Истощение CPU, диска и очереди обработки

**Риск: высокий.**

Для загрузки нет отдельного rate limit, ограничения одновременных загрузок,
лимита очереди и общего бюджета обработки. ZSTD level 22 использует все
доступные CPU-потоки через `threads=-1`, а обработчик выполняется из процесса
FastAPI через `asyncio.to_thread`
(`WEB-itinvent/backend/services/my_files_service.py:406`,
`WEB-itinvent/backend/services/my_files_service.py:414`,
`WEB-itinvent/backend/services/my_files_service.py:918`).

Обычная или скомпрометированная учётная запись может ухудшить работу всего API
повторными и параллельными загрузками.

**Что сделать:**

- добавить rate limit загрузок по пользователю и IP;
- ограничить число активных загрузок и задач обработки на пользователя;
- ограничить длину общей очереди;
- вынести обработку файлов в отдельный worker-процесс с лимитами CPU, памяти и
  времени;
- ограничить ZSTD до фиксированного числа потоков, например 2–4.

### 2. Quota race и запись полного тела до проверки квоты

**Риск: высокий.**

Тело запроса сначала полностью записывается в spool, и только затем вызывается
`create_pending_upload`, где проверяется пользовательская квота
(`WEB-itinvent/backend/api/v1/my_files.py:118`,
`WEB-itinvent/backend/api/v1/my_files.py:122`,
`WEB-itinvent/backend/services/my_files_service.py:237`,
`WEB-itinvent/backend/services/my_files_service.py:284`).

Параллельные запросы могут одновременно увидеть свободную квоту и превысить
лимит 5 GiB. Даже отклонённый после записи файл временно занимает диск.

**Что сделать:**

- резервировать квоту атомарно до приёма тела запроса;
- блокировать строку квоты пользователя или использовать advisory lock;
- учитывать активные upload-сессии и освобождать резерв при ошибке;
- сохранить текущий потоковый hard limit как дополнительную защиту.

### 3. Нет карантина и антивирусной проверки

**Риск: высокий для публичного файлового обмена.**

Любой активный пользователь может загрузить произвольный файл и создать
публичную ссылку. Перед статусом `ready` файл не проверяется антивирусом.
Дополнительно сервер обрабатывает недоверенные изображения и видео внешними
декодерами.

**Что сделать:**

- после загрузки помещать файл в карантин;
- проверять файл Microsoft Defender или ClamAV;
- разрешать share/download только после результата `clean`;
- хранить результат, версию сигнатур и причину блокировки;
- запускать media-обработку в изолированном worker-процессе.

## P2: исправить после P1

### 4. Публичный share-token хранится в открытом виде

**Риск: средний.**

В базе одновременно хранятся `share_token` и `share_token_hash`
(`WEB-itinvent/backend/appdb/models.py:362`,
`WEB-itinvent/backend/appdb/models.py:363`,
`WEB-itinvent/backend/services/my_files_service.py:675`).

Чтение базы или резервной копии раскрывает все активные публичные ссылки. Хеш
при наличии исходного токена почти не даёт защиты.

**Что сделать:** хранить только хеш и при запросе создавать новую ссылку либо
шифровать токен, если обязательна повторная выдача той же ссылки. Не писать
токены в access-логи.

### 5. Нет отдельных прав на загрузку и публикацию

**Риск: средний.**

Все endpoints используют только `get_current_active_user`, без отдельных
permissions (`WEB-itinvent/backend/api/v1/my_files.py:95`,
`WEB-itinvent/backend/api/v1/my_files.py:116`,
`WEB-itinvent/backend/api/v1/my_files.py:211`).

Любая активная учётная запись может использовать систему как публичный файловый
хостинг.

**Что сделать:** добавить серверные права `my_files.read`, `my_files.upload`,
`my_files.share` и административное право для расследования/блокировки.

### 6. Нет журнала действий

**Риск: средний.**

Для загрузки, создания и отзыва ссылки, скачивания и удаления отсутствует
append-only аудит. При инциденте нельзя надёжно установить действия пользователя
и обращения к публичной ссылке.

**Что сделать:** журналировать actor, file ID, действие, время, IP и user-agent.
Не сохранять сырой share-token и без необходимости полное имя файла.

### 7. Rate limit отключается для внутренних запросов

**Риск: средний.**

Общий guard пропускает внутренние запросы без ограничения
(`WEB-itinvent/backend/utils/rate_limit_guard.py:33`,
`WEB-itinvent/backend/utils/rate_limit_guard.py:43`).

Скомпрометированный компьютер внутренней сети сможет массово обращаться к
публичным ссылкам без прикладного throttling.

**Что сделать:** для файлового контура использовать повышенные внутренние
лимиты, но не полностью отключать rate limit.

### 8. Права Windows-каталога шире необходимых

**Риск: средний, операционный.**

Проверенный ACL `data/my_files` разрешает группе `BUILTIN\Пользователи` чтение,
создание и добавление файлов; несколько SID имеют `Modify`.

**Что сделать:** оставить доступ только сервисной учётной записи PM2,
`SYSTEM` и администраторам. После изменения проверить запуск worker и скачивание.

## P3: дополнительное усиление

- Добавить для публичной страницы и downloads заголовки `Cache-Control:
  no-store`, `Referrer-Policy: no-referrer` и подходящий CSP. Сейчас download
  явно задаёт только `Content-Disposition`
  (`WEB-itinvent/backend/api/v1/my_files.py:84`).
- Для cookie-auth state-changing запросов добавить проверку `Origin` /
  Fetch Metadata или CSRF-token. `SameSite=Strict` уже существенно снижает
  риск, поэтому это не P1.
- Зафиксировать решение по глобальной дедупликации. Дедупликация по SHA-256
  между разными пользователями создаёт небольшой side-channel по времени
  обработки: пользователь с известным кандидатом может предположить, что такой
  blob уже существует.
- Проверить редактирование access-логов IIS/Uvicorn: URL публичной ссылки
  является bearer-секретом и не должен попадать в долгоживущие логи.

## Рекомендуемый порядок работ

1. Rate limits, лимиты конкурентности, атомарное резервирование квоты и
   ограничение потоков ZSTD.
2. Отдельный worker и антивирусный карантин.
3. Сузить ACL каталога хранения.
4. Добавить permissions и аудит действий.
5. Убрать открытое хранение share-token и защитить логи/заголовки.

## Обновление P2/P3 от 2026-06-04

Закрыто и проверено:

- стабильные share-token шифруются отдельным `MY_FILES_SHARE_TOKEN_KEY`; в БД
  остаётся hash для публичного поиска, plaintext backfill завершён;
- добавлены серверные permissions `my_files.read`, `my_files.write`,
  `my_files.share`, `my_files.audit.read`;
- добавлен append-only журнал `app.my_file_audit` для загрузок, share,
  скачиваний, удаления, истечения срока и антивирусной проверки;
- rate limits публичных ссылок и download-grant больше не отключаются для
  внутренней сети;
- публичные metadata/download получают `no-store`, `no-referrer`, `nosniff`,
  а download также получает CSP `sandbox`;
- state-changing browser-запросы проверяют `Origin` и Fetch Metadata.

Остаётся выполнить операционно:

- применить ACL `data/my_files` из elevated PowerShell:
  `powershell -File scripts/harden-my-files-acl.ps1`;
- проверить и при необходимости редактировать IIS access-логи, чтобы bearer
  share-token из URL не сохранялся долговременно;
- принять отдельное решение по небольшому timing side-channel глобальной
  дедупликации между пользователями.

Отдельный остаточный риск frontend dependencies:

- `npm audit --omit=dev` сообщает `high` для `axios` и `moderate` для
  `react-router`; на 2026-06-04 npm не предлагает исправленных совместимых
  версий. Автоматический `npm audit fix --force` не применялся.
