# HUB-IT Android shell

Native Android wrapper for the existing HUB-IT React/Vite frontend.

## Goal

- Keep one UI source: `WEB-itinvent/frontend`.
- Package `WEB-itinvent/frontend/dist` into an Android APK through Capacitor.
- Keep login/session/API stability as the first real-device gate; FCM/deep-link code is now present but still needs Firebase config and device verification.

## Prerequisites

- Node.js compatible with the frontend build.
- **JDK 21** (`JAVA_HOME`, e.g. `C:\AndroidDev\jdk-21`) — Capacitor 8 / AGP require Java 21 for compilation.
- **Android SDK** (`ANDROID_HOME`, usually `%LOCALAPPDATA%\Android\Sdk`).
- Android SDK command-line tools on `PATH` (`sdkmanager`, `adb`).
- Capacitor dependencies installed in this folder with `npm install`.

### Windows Server / без winget

From the repository root (installs Microsoft OpenJDK 21 + SDK packages, sets user env vars):

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\install-android-devtools.ps1
```

Open a **new** PowerShell window, then verify:

```powershell
java -version
echo $env:JAVA_HOME
echo $env:ANDROID_HOME
adb version
```

**Android Studio** is optional for Gradle APK builds; install it only if you need the GUI emulator or SDK Manager UI.

## First setup

```powershell
cd mobile-android
npm install
npm run add:android
```

`android/` is generated by Capacitor and should be committed after the first successful `npm run add:android`, except for Gradle/build/local artifacts ignored in `mobile-android/.gitignore`.

## Debug build

From the repository root:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\build-android-apk.ps1 -Configuration Debug -ApiUrl https://hubit.zsgp.ru/api
```

The script builds the frontend with `VITE_PLATFORM=capacitor`, checks that release config does not use `server.url`, syncs Capacitor, and runs Gradle.

Current debug APK intentionally uses `server.url = https://hubit.zsgp.ru`.
That keeps the WebView origin equal to the production web origin, so the existing HttpOnly cookie auth and 2FA flow can work unchanged.
The packaged `dist` mode is blocked until backend auth supports the Capacitor local origin or a separate mobile token flow.

## Passkey / fingerprint login

Android fingerprint login in this app is WebAuthn/passkey, not a local-only biometric unlock. The backend verifies a real passkey credential, so Android WebView must expose `navigator.credentials`.

Implemented in the APK shell:

- `androidx.webkit:webkit` and `androidx.credentials` dependencies are included.
- `MainActivity` enables `WebSettingsCompat.setWebAuthenticationSupport(..., WEB_AUTHENTICATION_SUPPORT_FOR_APP)`.
- Android manifest includes `asset_statements` and a verified App Link for `https://hubit.zsgp.ru`.
- `WEB-itinvent/frontend/public/.well-known/assetlinks.json` contains the current debug APK signing certificate.

The file must be deployed by the web frontend and reachable exactly here:

```text
https://hubit.zsgp.ru/.well-known/assetlinks.json
```

It must return HTTP 200 with `Content-Type: application/json`, without 301/302 redirects. After each debug APK rebuild, verify the signing cert:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\print-apk-signing-cert.ps1
```

Current debug APK SHA-256 (Android default debug keystore on this build host):

```text

```

An older fingerprint is kept in `assetlinks.json` for compatibility across build machines:


When a release keystore is created, add the release SHA-256 fingerprint to `assetlinks.json`; otherwise passkeys can fail again on the release APK.

### Passkey troubleshooting (external network)

Passkey is **disabled on internal network** (`10.0.0.0/8`) by design. On **mobile internet** (not corp Wi-Fi) open in the phone browser:

```text
https://hubit.zsgp.ru/api/v1/auth/login-mode
```

Expected: `network_zone: external`, `biometric_login_enabled: true`. If `internal` on mobile data, IIS/proxy sends wrong client IP — check `X-Forwarded-For` in [web.config](../WEB-itinvent/frontend/public/web.config).

Server-side check from repo root:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check-passkey-backend.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\verify-android-passkey.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\print-apk-signing-cert.ps1
```

If `PublicKeyCredential` stays unavailable in WebView, the APK also exposes the `HubitPasskey` Capacitor plugin (Android Credential Manager) as a fallback for login and trusted-device registration.

**WebView check** (USB debugging): Chrome → `chrome://inspect` → HUB-IT WebView → Console on `/login`:

```javascript
({ secure: window.isSecureContext, pk: !!window.PublicKeyCredential, cred: !!navigator.credentials, origin: location.origin })
```

If `pk: false`, update Android System WebView / Google Play services and reinstall the latest debug APK.

**App Links:**

```bash
adb shell pm get-app-links ru.zsgp.hubit
```

Domain `hubit.zsgp.ru` should be `verified`.

**First-time passkey flow**

1. Login with password (+ 2FA if required).
2. Accept “remember device” / register passkey when prompted.
3. Log out and use passkey button — fingerprint is the system passkey UI, not a separate local-only login.

**«Passkey недоступен» сразу после регистрации устройства**

Обычно это гонка: кнопка passkey уже видна, а `PublicKeyCredential` в WebView появляется через ~1 с после открытия `/login`. APK ждёт `hubit:webauthn-ready` и показывает «Подготовка passkey», затем включает вход по отпечатку. Установите свежий debug APK и подождите 1–2 с на экране логина.

## Native features added

### Deep links

The APK now accepts both verified HTTPS links and custom `hubit://` links.

Examples:

```bash
adb shell am start -a android.intent.action.VIEW -d "hubit://dashboard" ru.zsgp.hubit
adb shell am start -a android.intent.action.VIEW -d "hubit://tasks/TASK-42" ru.zsgp.hubit
adb shell am start -a android.intent.action.VIEW -d "hubit://chat/CONVERSATION_ID?message=MESSAGE_ID" ru.zsgp.hubit
adb shell am start -a android.intent.action.VIEW -d "https://hubit.zsgp.ru/chat?conversation=CONVERSATION_ID" ru.zsgp.hubit
```

Frontend route mapping lives in `WEB-itinvent/frontend/src/lib/capacitorLinks.js`.

### FCM push

Implemented pieces:

- Android plugin: `@capacitor/push-notifications`.
- Runtime permission request for Android 13+.
- Android notification channel `hubit_default`.
- Native runtime guard: if `google-services.json` is not present in the APK build, FCM registration is skipped instead of calling Firebase and risking an Android process crash.
- Current debug builds without `google-services.json` also exclude the native `@capacitor/push-notifications` Gradle module after `cap sync`, so Firebase code is not packaged into the APK.
- Frontend token registration/deletion through `/api/v1/settings/notifications/native-push-token`.
- Backend native token table migration: `20260525_0034_native_push_tokens`.
- Backend FCM HTTP v1 sender using Firebase service-account credentials.
- Push tap routing through the same deep-link mapper.

Required external setup before push can work on a real device:

1. Create/select Firebase project.
2. Add Android app id `ru.zsgp.hubit`.
3. Put `google-services.json` into:

```text
mobile-android/android/app/google-services.json
```

4. Configure backend env:

```env
FCM_PROJECT_ID=your_firebase_project_id
FCM_SERVICE_ACCOUNT_FILE=C:\secure\hubit-firebase-service-account.json
# or FCM_SERVICE_ACCOUNT_JSON={...}
```

5. Run backend Alembic migrations and restart backend/chat push worker.

Without `google-services.json`, the app should still open and work as a WebView shell, but native FCM token registration remains disabled and the push plugin is not packaged.

### Camera, QR and files

The APK manifest declares `CAMERA` and the `HubitPermissions` Capacitor plugin requests runtime camera access before `html5-qrcode` starts `getUserMedia`. QR still uses the existing WebView scanner flow from `useDatabaseQrScanner.js`.

If the scanner shows permission denied:

1. Reinstall the latest debug APK.
2. Open the QR scanner again and tap **Allow** on the Android camera prompt.
3. If you previously denied it: **Settings → Apps → HUB-IT → Permissions → Camera → Allow**.

Real-device checks still required:

- QR scan opens camera and resolves equipment by inventory number.
- Chat file upload uses Android file picker and sends files.
- Mail/task attachment flows work from Android file picker.

### WebView fallback

If a React route crashes inside the native shell, the error screen now keeps reload and adds a system-browser fallback for the current URL.
